Service (RESTful --> Basics)

For all RESTful calls the service which should be invoked by the server is the first parameter after the base url (depending on your deployment) as follow:

http://featureserver.org/{service}/
Supported services can be found here.

Layer / typeName (RESTful --> Basics)

Single Layer (RESTful --> Basics --> Layer / typeName)

If you want to request a single layer, then it is possible to use the layer name after the {service} parameter: 

http://featureserver.org/{service}/{layer}/
or as parameter: 
http://featureserver.org/{service}/?typeName={value}
Name of the parameter is defined by the invoked service.

Multiple Layers (RESTful --> Basics --> Layer / typeName)

In every other case, means when requesting multiple layers at a time, the layers need to be defined in the appropriate parameter, e.g. for service WFS 1.1.0 it is called typeName and WFS 2.0.0 it is typeNames followed by a comma seperated list.

OutputFormat (RESTful --> Basics)

Using RESTful API, the output format is defined by the 'requested file extension' in the url like the following example: 

http://featureserver.org/{service}/{layer}/{id}.{outputFormat}?{parameters}
or as parameter called outputFormat 
http://featureserver.org/{service}/{layer}/{id}?outputFormat={value}
Supported output formats can be found here.

WFS (RESTful)

independent A parameter applied to all feature types listed in the reuqest. In the following example the boudning box is independent and would therefore be applied to both layers, means all returned features are inside the given BBOX. 

typename=FEAT1,FEAT2&BBOX=5.9559,45.8180,10.4920,47.8084

dependent Parameters consisting of lists shall use the comma (",") as the delimiter between items in the list. In addition, multiple lists can be specified as the value of a parameter by enclosing each list in parentheses; "(", ")". The characters ",", "(" and ")" may be escaped using the "\" character. An example look likes this: 

typename=FEAT1,FEAT2&filter=(... FEAT1 filter...)(... FEAT2 filter...)

Common (RESTful --> WFS)

The following parameters can be used to define the request and are defined by OGC.

ParameterDescription
service
optional		 Defines which service should be invoked. If you invoke the service like described above (e.g. http://featuresever.org/{service}/), then this parameter is optional.
request
required		 The following values are allowed:
Values:[ GetCapabilities | DescribeFeatureType | GetFeature ]

Special keywords exists for the different requests types. See below.
version
required		 For WFS 1.x only the following value is allowed:
Value:[ 1.1.0 | 2.0.0 ]
filter
optional		 A OGC Filter Encoding compliant XML.
Applies to: the index of the feature type in the list dependent
Supported by: SpatialLite PostGIS PostGISHstore
bbox
optional		 A bounding box (BBOX) applies to all feature types listed in the request.
Format: lcc1,lcc2,...,lccN,ucc1,ucc2,...uccN
where lcc means Lower Corner Coordinate, ucc means Upper Corner Coordinate
Applies to: all feature types listed in the request independent
Supported by:
sortby
optional		 The sortby parameter is used to specify a list of property names whose values should be used to order (upon presentation) the set of feature instances that satisfy the query. The value of the sortby parameter shall have the form PropertyName [ASC|DESC][,PropertyName [ASC|DESC],...] where the letters ASC are used to indicate an ascending sort and the letters DESC are used to indicate a descending sort. If neither ASC nor DESC are specified, the default sort order shall be ascending.
Format: PropertyName [ASC|DESC][,PropertyName [ASC|DESC],...]
Applies to: all feature types listed in the request independent
Supported by:

Special (RESTful --> WFS)

These parameters are special parameters and not definded by OGC WFS standard.

ParameterDescription

optional		 Applies ordering on the result set, means on all requested data sources.

optional		 List of attributes to be ordered by. Ordering is important and would be applied to the result set, means on all requested data sources, thus the list is the least common denominator of all attributes in every data source.
Value:comma seperated list
Applies to: the index of the feature type in the list dependent

Keywords (Special Parameters)

Some keywords exist for getting the server's capabilities, describing layers or getting features.

KeywordDescription
capabilities Is used to get the capabilities of a service. It can be used as follow: 
http://featureserver.org/{service}/capabilities.{outputFormat}
This keyword replaces request=GetCapabilities
describe Use it to get a description of a single or multiple layers. To get the describption of a single layer you can use the following format: 
http://featureserver.org/{service}/{layer}/describe.{outputFormat}
and for multiple layers use id as follow: 
http://featureserver.org/{service}/describe.{outputFormat}?typeName={value}
where {value} is a comma seperated list of values and the parameter name typeName depends on the invoked service and version.

This keyword replaces request=DescribeFeatureType
features As described you can use the feature's id as filename to request its information as follow: 
http://featureserver.org/{service}/{layer}/{id}.{outputFormat}
In addition it is also possible to request a bunch of features using the keyword features instead of the id: 
http://featureserver.org/{service}/{layer}/features.{outputFormat}
In the above example a specific layer would be queried. For multiple layers you can define the layers a parameters 
http://featureserver.org/{service}/features.{outputFormat}?typeName={value}
where {value} is a comma seperated list and typeName depends on invoked service and version.

This keyword replaces request=GetFeature
